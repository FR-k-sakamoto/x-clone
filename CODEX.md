# CODEX.md

## 目的
X(旧Twitter)のクローンを構築する。コア体験は「読む・投稿する・反応する」。

## スコープ(必須)
- タイムライン閲覧(最新順/フォローのみ)
- 投稿作成(本文・文字数制限)
- いいね、リポスト(再投稿)
- 返信(スレッド表示)
- プロフィール表示
- 認証(サインアップ/ログイン/ログアウト)

## 非スコープ(当面やらない)
- DM(ダイレクトメッセージ)
- 通知センターの高度機能(おすすめ通知など)
- 広告配信・課金
- 高度な検索/トレンド

## ユーザー体験要件
- 主要操作は3クリック以内で到達できる
- 投稿作成は即時にUIへ反映(楽観的UI)
- タイムラインはスクロールで追従読み込み(無限スクロール)
- モバイル/デスクトップ両対応

## 機能要件
- 必須
  - ユーザー登録・ログイン
    - Google認証のみ
    - セッション維持
  - 投稿の作成・編集・削除
    - 文字数上限: 160
  - タイムライン表示
    - 最新順
    - フォローのみのフィルタ
  - いいね
  - リポスト
    - 元投稿を参照
  - 返信(リプライ)
    - 親投稿の参照を保持
  - フォロー/フォロワー
  - 検索(投稿・ユーザー)
  - プロフィールの表示・編集
- 任意(発展)
  - ハッシュタグ
  - ブックマーク・ピン留め
  - 通知
  - 非公開アカウント(フォロー承認)
  - 画像投稿

## 非機能要件
- パフォーマンス: 初期表示2秒以内(ローカル開発基準)
- 可用性: 主要エラーはUIで説明
- セキュリティ: XSS/CSRFを考慮し入力をサニタイズ
- 監視: 重要操作のログ(投稿/いいね/フォロー)

## 技術方針(現在のリポジトリ前提)
- Next.js App Router
- TypeScript
- ESLint/Prettier準拠
- デプロイ: Render
- BaaS: Supabase
  - DB: Supabase Postgres
- ORM: Prisma
- Auth: NextAuth + bcrypt
- UI: MUI(icons含む)

## 作業ルール
- 変更は小さく、意図を明確に
- 既存構造を尊重し、無断で大規模リファクタしない
- UI変更はスクリーンショット/説明を付ける
- 追加依存は理由を明記する
- 仕様変更はこのドキュメントを更新する
- Phase単位で作業を進めること
- Phase3 はドメインごとに作業を進めること
- 作業が完了したら確認依頼をし、承認をもらったうえで `docs/PHASES.md` のチェックボックスにチェックをつけること

### 1. ワークフロー: Git Worktree
- ドメイン(機能単位)ごとに Worktree を作成し、並列開発を行う
- Worktree 間で依存関係が生じる場合は、共通ドメインのインターフェースを先に定義すること
- ブランチ戦略は以下に統一する
- `main`: デプロイ用
- `develop`: 開発用
- `feature/{domain-name}/{task}`: 作業用
- `feature/{domain-name}/{task}` は必ず `develop` から派生させる

### 2. アーキテクチャ: クリーンアーキテクチャ(ディレクトリ構造)
Next.js の App Router 構造を意識しつつ、ロジックを分離する。

- `/src/domain`: 最重要。純粋な TypeScript クラス/型
- Entity、ValueObject、Repository Interface を置く
- React や Next.js の機能(Hooks、API)を絶対にインポートしない
- `/src/application`: ユースケース(UserStories)
- Domain オブジェクトを操作する関数やクラスを置く
- `/src/infrastructure`: 外部接続の具体実装
- API クライアント(fetch/axios)、Firebase、LocalStorage、リポジトリ実装を置く
- `/src/components`: プレゼンテーション
- UI 構築のみを担当し、ドメインロジックを書かない
- `/src/hooks`: アプリケーション層と UI 層の橋渡し

### 3. 実装方針: オブジェクト指向(OOP)
React の関数型コンポーネントの中で、あえてオブジェクト指向を徹底する。

#### ドメインモデルの徹底
- Anemic Domain Model(貧血ドメインモデル)は禁止する
- 単なる interface(データだけの箱)をコンポーネント間で回さない
- ロジックを持つ Class として定義し、不変条件のチェック(バリデーション)や計算処理をメソッドとしてカプセル化する
- 手続き型 Service は避ける
- `UserService.update(user)` ではなく `user.changeEmail(newEmail)` のように、オブジェクト自身に振る舞いを記述する

#### React と OOP の共存
- コンポーネント内では、State としてドメインインスタンスを保持するか、プレーンなオブジェクトからドメインインスタンスを再生成してメソッドを呼び出す
- 例: `const price = new Order(orderData).calculateTotal();`

### 4. Next.js 固有のルール
- Server Actions は Application Layer のエントリーポイントとして扱う
- RSC(Server Components)は Infrastructure Layer からデータを取得し、Domain インスタンスを生成して Client Component へ渡す役割を担う

## ディレクトリ/命名ルール
- `app/`配下はルーティングに合わせる
- UIは`components/`、ロジックは`lib/`
- 型は`types/`、定数は`constants/`
- 1ファイル1責務を基本

## 受け入れ基準(簡易)
- 新規投稿がタイムラインに表示される
- 返信がスレッドとして表示される
- いいね/リポストが反映される
- ログインしていない場合は投稿不可
